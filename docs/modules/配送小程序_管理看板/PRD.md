# 广交会项目 - 配送小程序管理看板 PRD（评审版）

> 版本：V0.1  
> 日期：2026-03-01  
> 模块：配送微信小程序 / 配送管理看板  
> 关联文档：`运营配送方案.md`

## 1. 业务背景

广交会现场配送业务采用“柔性调度、主辅结合”的灵活管理模式，要求赋予前线配送管理员（或区长、督导）全局的巡查能力。
为此，必须在“配送微信小程序”中提供一个核心的“管理看板”模块，帮助管理员在移动端随时随地掌握各区履约进度并进行跨区干预。

## 2. 核心体验原则（战前/战中/战后 视角）

1. **扁平化权限，全域可见**：登入小程序的主管，对全馆各防区拥有通查权限。
2. **本地视图记忆**：通过 `LocalStorage` 记住主管最后盯防的切片（如 C区），冷热启动无感聚焦。
3. **时间轴批次瀑布流排布（核心）**：打破常规数据仪表盘大平层，界面主体采用**各个配送时段批次（如 10:30 批次、11:00 批次、宵夜特殊批次）**首尾相连组成纵向卡片瀑布流。通过“进行中”、“未开始”、“已结束”三种折叠态，为主管展现过去、现在和未来的战斗压力。
4. **视觉极简与下钻深潜**：
   - 表层卡片使用极简 UI 组件（以“悠饭橙”为主色调，不用蓝色等冲突色），只露血槽（进度）和警报（超时数字）。
   - 里层增加独立的下钻明细页（如：超时堵点详情跳转页），在明细里罗列到人、到点位、到商家的具体责任。

## 3. 目标用户与角色

- **配送主管 / 区长 / 大巡查督导**：在移动中实时查看自身防区的压力情况，或切到全局视图查看全局水位，以便进行跨区运力调度借调。手机小程序是其核心作战台。

## 4. 核心功能需求（FR）

### 4.2 批次时间轴视图（页面中枢）

*页面主体由上至下按“紧急程度与执行期”排列的动态履约批次卡片流。*
**批次命名范式**：强制采用 `[截单时间]~[清货时间]` 进行命名。共包含 5 个明确批次：`9:15~10:00`, `9:45~10:30`, `10:15~11:00`, `10:45~11:30`, 以及特殊的连轴段 `10:46~14:00`（第4批次截单后即刻开启）。

**批次卡片呈现与排序规则（严格按以下 4 种类型降序排列）**：

#### 4.2.1 【最高危置顶】类型 2：已过清货时间，但仍未结案的批次
- **触发条件**：当前时间 > 批次规定的清货时间，但 `remainCount > 0`。宵夜餐段因清货时间设置到 23:00，不会出现此状态。
- **渲染策略**：纯白底色 + 加粗红边框 + 卡片顶部红线（`border-t-4`）+ 标题飘红字色。
- **展示内容（五模块结构）**：
  1. 时段展示：居左 `[截单时间]~[清货时间]`，居右状态标签“配送超时”
  2. 订单总量与送达率：居左“总需配送 XX 单”，居右“已送达/完成率 XX/XX%”。完成率 = 总送达量 / 总下单量 * 100%
  3. 送达流速与完成预估（计算公式见下方附录）
  4. 实时配送流转节点（四段漏斗，结构见下方附录）
  5. 实时堵点预警榜（五种预警类型，见下方预警字典）

#### 4.2.2 【主战区】类型 1：进行中的批次
- **触发条件**：当前时间 介于 `截单时间` 与 `清货时间` 之间。
- **渲染策略**：自动定位、默认全展开。
- **展示内容（五模块结构）**：
  1. 时段展示：居左 `[截单时间]~[清货时间]`，居右状态标签“进行中”
  2. 订单总量与送达率：居左“总需配送 XX 单”，居右“已送达/完成率 XX/XX%”
  3. 送达流速与完成预估（计算公式见下方附录）
  4. 实时配送流转节点（四段漏斗，结构见下方附录）
  5. 实时堵点预警榜（五种预警类型，见下方预警字典）
- **宵夜/动态批次特殊规则**（`10:46~14:00`）：
  - 第二模块替换为：居左“当前下单总量 XX”，居右“已出单/已送达 XX/XX”
  - 第三模块（送达流速）不显示
  - 第五模块中不出现 `KITCHEN_DELAY` 以外的出单超时（宵夜餐段独有出单超时预警）

***附：📐 实时配送流转漏斗 - 核心指标计算公式栈***
*为杜绝前后端认知分歧，漏斗内四大节点（待接单、赴集散、在集散、送展位）的数据源口径如下定义：*
1. **[袋数与订单量]**：处于该节点状态的关联打包编织袋总数（袋）及订单总数（单）。*注意：物理世界中一袋可装多单，因此袋数通常少于单量，UI 排版必须遵循「袋数在上，单量在下」的原则，如 45袋 / 120单*。
2. **[节点当前挂载人数]（活体运力）**：从上述处于该节点的订单组合中，提取其绑定的 `courier_id` 或 `signer_id`，并进行**去重 (Distinct)**。*(注：非历史累计，仅统计此时此刻正背着这批货的人)*。
3. **[四段均耗时间]（段跃迁测速，单位: 分钟）**：
   - 采样边界：取当前批次内，成功走通对应截面（完成状态跃迁）的订单集合。
   - **待接单均耗** = `Avg(配送员在商家扫码接单时间 - 出单时间)`
   - **赴集散均耗** = `Avg(分拣签收员在集散地扫码签收时间 - 配送员在商家扫码接单时间)`
   - **在集散均耗** = `Avg(配送员在集散地扫码接单时间 - 分拣签收员在集散地扫码签收时间)`
   - **送展位均耗** = `Avg(配送员送达完成时间 - 配送员在集散地扫码接单时间)`

***附：📈 送达流速与完成预估 - 计算公式***
1. **送达流速**（单位: 单/分）= 该批次送达时间在近 5 分钟内的订单数量 / 5
2. **预估送完还需**（单位: 分钟）= 当前该批次未送达单量 / 送达流速。值小于 1 时取 1；送达流速为 0 时显示“未知”
3. **不显示条件**：该批次无已送达状态订单时不显示；该批次订单全部送达时不显示
4. **宵夜餐段**：不显示送达流速与完成预估模块

***附：🚨 实时堵点预警字典（现场核心干预依据）***
*当流转停滞时间超过设定的阈值，将自动在“进行中”及“烂尾”批次卡片下方触发警铃榜单，并在下钻页支持电话穿透调度：*
1. **[KITCHEN_DELAY] 出单超时**：仅宵夜餐段存在。用户下单后商家超 30 分钟未出单。展示最长滞留时间与超时出单数量。下钻页以商家为单位，按最长滞留时间倒序排序。（干预：催后厨/停接单）
2. **[NO_RIDER_ACCEPT] 待接单超时**：出单后超 10 分钟无配送员扫码接单。展示最长滞留时间与待接单数量。下钻页以商家为单位，按最长滞留时间倒序排序。（干预：调游离运力抢单）
3. **[GO_HUB_TIMEOUT] 送集散地超时**：配送员在商家扫码接单后，超 8 分钟未送到集散地。展示最长滞留时间与超时订单数量。下钻页以帐号为单位，按最长滞留时间倒序排序，含帐号姓名、手机号及配送单元明细（展位号+袋数+订单数+滞留时间）。（干预：催配送员加速）
4. **[HUB_STAY] 在集散地滞留超时**：订单在集散地签收后，超 7 分钟未被配送员接单。展示最长滞留时间与滞留超时订单数量。下钻页以展厅点位为单位，按最长滞留时间倒序排序。（干预：催站长清场）
5. **[DELIVERY_TIMEOUT] 送展位超时**：配送员在集散地扫码接单后，超 25 分钟仍未送达。展示最长滞留时间与超时订单数量。下钻页以帐号为单位，含帐号姓名、手机号及配送单元明细（展位号+袋数+订单数+滞留时间）。（干预：催配送员）

#### 4.2.3 【防守区】类型 4：未开始批次
- **触发条件**：当前时间 < 批次截单时间。
- **渲染策略**：半折叠态，灰色/禁用质感。
- **展示内容**：居左 `[截单时间]~[清货时间]` + “未开始”，居右“已下单量 XX 单”。已下单量 = 当前该餐段已下单量（排除已取消订单）。宵夜餐段特殊：居右展示“未开放下单”。

#### 4.2.4 【锚点区】类型 3：已完成批次
- **触发条件**：批次内的单量全部送达，或已关账。宵夜餐段特殊：截单后配送单元都已送达即为已完成。
- **渲染策略**：极简折叠态，排在列表最末端，按餐段清货时间倒序排列。
- **展示内容**：居左 `[截单时间]~[清货时间]` + “已完成”，居右“已完成 XX 单”。

## 5. 关键交互要求

1. **流畅的顶部切换**：展区横向 Tab 切换必须是即时反馈的，带有平滑的下划线游标动画，不可有极长白屏阻塞。可在请求数据期间骨架屏占位。
2. **沉浸式硬件交互**：“扫码放行”抽屉推拉顺滑，开启后自动唤起硬件摄像头并覆盖全屏，配合取景框 UI 与声音反馈，支撑高频连扫作业场景。
3. **极简降噪**：界面不要有过深的折叠菜单或下钻图表。以大号数字、进度条、红绿黄状态灯（正常/警告/严重）为主，适合管理员“边走边看”。
4. **防误触下拉拉齐**：支持全局下拉刷新，因为对于现场调度而言，数据的新鲜度至关重要。

## 6. 与「PC统计看板」的边界划分

- **PC 配送统计页**：适合在指挥中心大屏，进行细致到每一张订单/每个责任人的多维穿透、下钻与名单导出。
- **小程序配送看板**：聚焦于现场调兵遣将所需的核心“卡口”。只输出效率劣势靶面（某区某商家慢了、哪一环均值崩了），以及即时报警的 Top 堵点。不提供单件列表下钻，发现问题后直接执行对讲机线下调度。

## 7. 异常处理流

- **弱网环境**：接口拉取失败时提示“数据刷新失败，请轻触重试”，避免白屏死锁，同时若有旧缓存数据可灰态打底展示。
- **无权限强跳**：若无管理员角色的普通配送员通过非法手段进入此页面（理论不应出现），显示全局缺省页“暂无查看权限”。
