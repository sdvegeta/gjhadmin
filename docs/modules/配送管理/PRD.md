# 广交会项目 - 后台配送管理界面 PRD（评审版）

> 版本：V0.4（原型功能完全对齐版）  
> 日期：2026-03-01  
> 适用范围：运营后台「配送管理界面」子项目（不含配送统计页、不含配送小程序筛选页）  
> 产出目标：为高保真原型与研发拆解提供一致业务基线

## 0. V0.4 原型对齐决策（路径A+B）

1. 原“主责覆盖查阅”与“主责覆盖配置”已彻底合并为单一的 `配送员主责配置` Tab，功能高度聚合，提升空间利用率和操作体验。
2. 责任配置台移除原先的顶部前端伪权限分层视角。在筛选项中保留展区、楼层、展厅外，新增了 **主责商家名称模糊查** 的筛选项。
3. 配送员名册增强检索：不仅支持 `姓名/手机号` 查询，新增**角色快筛**与**帐号状态（启用/停用）快筛**。
4. 增强角色体系（路线B）：引入独立【配置角色】弹窗，且**人员姓名直接正常文本显示**，不使用任何禁用表单件。单人可多角色重叠，且明确【分拣签收员】由于拥有全域权限，**不可且无需配置任何点位/商家**。
5. 账号生命周期由硬删除（抹除）改为软删除（停用/恢复）。当新增人员手机号与已停用库中账号冲突时，弹窗提示并支持一键原地唤醒更名。
6. 取消前端风险数字强制展示，取消表格列表的双重内层滚动条约束，采用原生流伸展。
7. “配置展厅点位”弹窗大改：完全抛弃旧有的“选展区=全管辖”复杂继承推导和警告提示。重构为彻底的“合理利用空间、全盘显性平铺选择”交互体验。
8. 商家 Mock 数据统一：去除“M001”等丑陋占位符，配置商家弹窗不再保存独立区域字段，统一采用直接拼接到名称的方式，如：“肯德基（A区）”。

## 1. 业务背景

广交会场景下，配送组织采用“同区履约 + 分组负责 + 分段配送”主干逻辑。为保证峰值期可控履约，需要在运营后台提供统一的配送配置入口，解决以下问题：

1. 点位与商家供应关系需可配置，避免点位出现无可服务商家或商家跨区错配。
2. 配送员账号需快速创建与维护，满足大促期临时扩编。
3. 配送员“主负责范围”需结构化配置（展区/楼层/展厅），并支持批量配置。
4. 运营与区长需按展区-楼层-展厅快速查询当前主责任配送人员。

## 2. 产品目标与非目标

### 2.1 产品目标

1. 建立点位-商家映射能力，支撑排期时点位只显示可服务商家。
2. 建立配送员账号管理能力，手机号唯一，账号创建流程最简。
3. 建立“主负责范围”配置能力，支持多人批量配置与层级继承规则。
4. 建立主责查询能力，按展区/楼层/展厅可视化查询责任人。

### 2.2 非目标（本期不做）

1. 不包含配送统计页建设。
2. 不包含配送小程序筛选模块改造。
3. 不包含自动派单算法与实时调度策略改造。
4. 不包含跨区支援策略自动化（仅保留规则字段与人工扩展能力）。

## 3. 目标用户与角色

1. 运营配置人员：维护点位-商家关系、配送员账号、主负责范围。
2. 区长/配送主管：按区域查看主责人、检查配置完整性。
3. 超级管理员：全局配置与审计追踪。

## 4. 核心业务规则

### 4.1 点位-商家映射规则

1. 一个点位可绑定多个商家。
2. 同一点位禁止重复绑定同一商家。
3. 排期/下游使用点位时，仅返回该点位已绑定商家。

### 4.2 配送员账号规则

1. 新增配送员仅需姓名、手机号，默认初始角色为【配送员】。
2. 手机号为全局唯一键，重复手机号禁止保存。但如遇已停用的手机号重新入职：支持检测并原地弹窗激活（即软删除唤醒替换）。
3. 提供“停用”和“恢复”机制控制账号活性。
4. 提供“配置角色”能力。支持【配送员】【分拣签收员】【配送主管】三种枚举。
5. 角色分配防呆规则：必须保留至少1个角色；【配送员】与【分拣签收员】身份绝不可共存；【配送主管】可与任一角色共存。
6. **全局权限例外**：【分拣签收员】对所有展厅点位和商家天生具备全部操作权限，因此在配置台该类人员自动被剔除（不可选中），无需为其配置展厅或商家。

### 4.3 主负责范围配置规则

1. 配置维度为：展区-楼层-展厅。界面要求合理利用空间，保证所有楼层和展厅均为**显性平铺选择**，不折叠、不利用下拉等隐性控件。
2. 三层均支持多选，支持批量选人后一次配置。
3. **取消旧版的层级继承覆盖逻辑**（即取消“只选展区代表全管辖”及配置界面的警告提示）。改为纯粹的所见即所得、选哪里就是哪里的离散显性绑定。
4. 同一配送员可配置多条主责范围记录。
5. 系统保存时将“显式选择的最终层级节点”精准标准化落库。

### 4.4 主责查询规则

1. 支持按展区、楼层、展厅、商家名称快速筛选查询主责任配送员。
2. 查询结果在 `配送员主责配置` 的统一列表中展示，支持直观显示对应配送员的：**姓名、手机号、角色、主责展厅点位、主责商家**。

## 5. 页面范围（后台配送管理界面）

1. 点位-商家配置页。
2. 配送员账号管理页。
3. 配送员主责配置（合并主负责范围展示查阅 + 批量主责点位及商家配置能力）。

## 7. 关键交互要求（用于高保真原型）

1. 页面采用 Tab 分区，避免单页信息过载。
2. 统一三层筛选器：展区 -> 楼层 -> 展厅，选项同级联动。点位商家配置Tab需单独增加提示：“此处展厅点位排期仅能选择此处配置的对应商家。”。
3. 独立且优化的展示控件：取消表格内部滚动条限制；数据编辑弹窗中仅待选项提供搜索/编辑控件，非编辑基准信息（如配送员名册的角色弹窗内的人员**姓名正常直接显示，不要采用禁用输入框或其他奇怪组件**）；角色配置由以往的复选框改为**带有副标题的卡片组件**增强指引。超50+项长列表（如商家池多选）支持文本快捷筛选查库以提升效能。
4. `配送员主责配置` 台采用同屏配置查询交互，支持：
   - 过滤身份为【分拣签收员】的账号，置灰多选框不可选（其自带全局权限）；包含该角色的行甚至可以在某些操作时禁用响应。
   - 提供表格上方单独的【配置展厅点位】和【配置责任商家】独立按钮与对应弹窗。
   - 移除繁杂的“覆盖提示文案”与内部树枝干推导，配置展厅点位的弹窗全部铺开，“显性选择”，只平铺展示所有待选点位，取消任何包含/继承警告文案；
   - 列表假数据全域对齐，剔除例如 `M001` 等技术向 Mock 占位符。商家名称直出格式要求拼上地区后缀（不走独立字段），如 `肯德基（A区）`。由于系统暂未存商家和区域的强对应关系，候选列表中须剔除类似 `[A区]` 的前置区域分类标识。
5. 手机号重复时前端即时提示，检测若是历史失活号即弹窗引导一键激活更新信息。

## 8. 数据指标（首期）

1. 点位商家配置完成率（已配置点位数 / 总点位数）。
2. 配送员主责配置覆盖率（已配置主责配送员数 / 启用配送员数）。
3. 配置错误拦截率（重复手机号、重复映射、非法作用域）。
4. 查询命中时延（P95）。

## 9. 验收标准

1. 新增配送员时，重复手机号被阻断，且提示明确。
2. 批量配置主责时，能正确生成 `AREA/FLOOR/HALL` 作用域记录。
3. 查询任一展厅时，能返回展厅级 + 楼层级 + 展区级责任人。
4. 点位绑定商家后，下游排期接口仅返回已绑定商家。
5. 所有增删改操作都有审计日志。

## 10. 风险与应对

| 风险 | 影响 | 应对策略 |
|---|---|---|
| 展馆字典变更频繁 | 配置失效或漏配 | 字典接口按版本号拉取，配置表保留软引用与快照 |
| 批量配置误操作 | 大范围错误覆盖 | 提交前二次确认 + 变更预览 + 审计回滚 |
| 手机号复用或错填 | 账号冲突 | 前后端双重唯一校验 + 删除二次确认 |
| 主责继承理解偏差 | 查询结果歧义 | 结果页展示命中层级与规则说明 |
